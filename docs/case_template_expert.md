# Шаблон кейса: экспертное интервью

Шаблон для кейсов, собранных из интервью со спикером (один продукт / одна команда / одна DS). Для обзорных кейсов — отдельный шаблон.

## Как пользоваться

1. Скопируй всё ниже в новый файл `Кейс_<Компания>_<DS>.md`.
2. Удаляй подсказки в `<!-- -->` по мере заполнения.
3. Заполняй сверху вниз: шапка → паспорт → TL;DR → история → хвост. Не начинай с истории — без паспорта не поймёшь, что вытаскивать.
4. Держи под рукой памятку по стилю (в конце файла). Перечитывай её перед каждым разделом.
5. Когда кейс готов — переложи по полям в сидер (см. «Маппинг на БД» в конце).

## Памятка по стилю

Перечитай перед публикацией. Если хоть один пункт нарушается — переписывай.

1. **Никаких зачинов «Кейс подчёркивает / фиксирует / описывает / показывает».** Начинай с факта, числа, сцены или цитаты.
2. **Одна мысль — один абзац.** Абзац длиннее 4 строк — разбить.
3. **Цифры вместо оценок.** Не «много команд» — «к концу 2023 года подключилось ~70 команд, затем >130».
4. **Цитаты вплетены, не свалены.** В историю — по одной на раздел, как иллюстрация. В хвост — отдельной подборкой.
5. **Списки и таблицы — когда это реально список.** Процессы, ритмы, стек, роли — таблица. История и инсайты — живой текст.
6. **Активные глаголы, настоящее время.** «Команда собирает», а не «командой собирается».
7. **Длина раздела пропорциональна его весу в кейсе.** Не выравнивай объёмы ради симметрии.
8. **Проверка на «универсальность»:** если абзац можно вставить в любой другой кейс без изменений — он пустой. Переписать с конкретикой.
9. **Каждая «фишка» в разделе 4.5 должна помещаться в твит.** Если не помещается — это не фишка, а описание процесса.
10. **TL;DR пишется последним.** Сначала — вся история, потом — выжимка из неё.
